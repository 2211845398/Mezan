## 1. الفكرة العامة (The Core Concept)

### 1.1 المشكلة التي يعالجها المشروع

في بيئات التجزئة متعددة الفروع، تتشتت البيانات التشغيلية (المبيعات، المخزون، المشتريات، الموارد البشرية) بين أنظمة منفصلة أو سجلات يدوية، ما يُضعف دقة القوائم المالية ويزيد زمن إقفال الحسابات. يهدف **Mezan** إلى معالجة هذا الانفصال عبر منصة موحّدة تجمع **العمليات اليومية** و**المحاسبة المالية** في مسار بيانات واحد.

### 1.2 الفكرة المحورية

**Mezan** نظام تخطيط موارد مؤسسات (ERP) وإدارة تجزئة، يُبنى حول محورين متكاملين:

1. **محرك نقاط البيع (POS)** — واجهة التنفيذ اليومي للمبيعات والتحصيل.
2. **نواة المحاسبة بالقيد المزدوج** — دفتر أستاذ عام (GL) يُغذّى آلياً من الأحداث التشغيلية.

### 1.3 آلية الربط بين العمليات والمحاسبة

- **من التشغيل إلى القيد:** كل حدث تشغيلي مؤكّد — إتمام بيع، إرجاع، استلام بضاعة، صرف راتب، تعديل مخزون، سند قبض/صرف — يمر عبر طبقة خدمات (`services/`) تُطبّق قواعد الأعمال ثم تُطلق **ترحيلاً محاسبياً آلياً** إلى GL عبر خدمات الترحيل (`document_posting_service` وما يتبعها).
- **التحقق المحاسبي:** لا يُقبل أي قيد إلا إذا تحقق **توازن المدين والدائن**، وارتبط بفرع (`branch_id`)، واحترم حالة **الفترة المالية** (مفتوحة / إقفال مرن / مغلقة).
- **التكامل العكسي:** التقارير المالية (ميزان المراجعة، قائمة الدخل، الميزانية العمومية، كشوف AR/AP) تُستخرج من نفس مصدر الحقيقة الذي أنتجته العمليات، دون إعادة إدخال يدوي.
- **النتيجة التحليلية:** النظام لا يُقدّم «وحدة محاسبة منفصلة» و«وحدة مبيعات منفصلة»، بل **نموذج بيانات موحّد** يفرض اتساقاً بين الرقم التشغيلي والرقم المالي من لحظة الحدوث.

### 1.4 القيمة الهندسية للتصميم

- **فصل الطبقات:** مسارات API للتحقق والتوصيل فقط؛ المنطق التجاري في الخدمات؛ النماذج والمخططات (Schemas) للبيانات — ما يسهّل الاختبار والصيانة.
- **تعدد الفروع كأولوية:** المخزون، القيود، وورديات POS مرتبطة بفرع محدد، مع إمكانية التجميع والتفصيل في التقارير.
- **واجهة عربية أولاً (RTL):** النظام موجّه لبيئة محلية، مع دعم ثنائي اللغة (عربي / إنجليزي).

## 2. النطاق (Scope)

### 2.1 الجمهور المستهدف

محلات **التجزئة متعددة الفروع** — مزيج من فروع بيع (متاجر) ومستودعات — التي تحتاج:

- نقطة بيع سريعة وموثوقة.
- رؤية مخزون لحظية عبر الفروع.
- محاسبة مالية متوافقة مع العمليات دون تأخير إقفال.

### 2.2 الوحدات المنفَّذة فعلياً

#### أ) العمليات (Operations — Ops)

| الوحدة | أبرز القدرات المنفَّذة |
|--------|-------------------------|
| **نقاط البيع (POS)** | ورديات، عربات، خصومات، طرق دفع متعددة، فواتير مبيعات، إرجاعات وتبديلات، إيصالات |
| **الكتالوج** | فئات هرمية، منتجات، متغيرات (Variants)، وحدات قياس، ضرائب، قوائم أسعار |
| **المخزون** | مستويات لكل فرع، حجز وفي الطريق، تحويلات بين الفروع، تعديلات، تنبيهات إعادة الطلب |
| **المشتريات** | أوامر شراء، استلام بضاعة، مسح فواتير (OCR)، كشف حساب مورد |

#### ب) المالية (Finance)

| الوحدة | أبرز القدرات المنفَّذة |
|--------|-------------------------|
| **المحاسبة** | دليل حسابات هرمي، قيود يومية، فترات مالية، عكس قيود |
| **الذمم المدينة والدائنة** | AR/AP، سندات قبض وصرف، أرصدة افتتاحية |
| **التقارير** | ميزان مراجعة، دفتر أستاذ، قائمة دخل، ميزانية عمومية، تقارير حسب الفرع |
| **العملات** | عملات متعددة على مستوى القيود، إعادة تقييم فروقات الصرف عند إقفال الفترة |
| **تقييم المخزون** | متوسط مرجّح (WAVG) أو FIFO حسب سياسة المنشأة |

#### ج) الموارد البشرية (People)

| الوحدة | أبرز القدرات المنفَّذة |
|--------|-------------------------|
| **الموظفون** | ملفات، جداول عمل، حضور وانصراف، طلبات إجازة |
| **الرواتب** | احتساب شهري، مسيرات، موافقة وصرف، تصدير CSV للبنك |
| **الانضمام** | تدفق IT → HR → تفعيل الحساب |

#### د) الدعم والتحليل

- **CRM والتسويق:** ولاء، خصومات، أداء العملاء.
- **ذكاء الأعمال:** لوحة تنفيذية (KPIs) حسب الدور.
- **مستشارون ذكيون:** دعم قرارات في المشتريات والموارد البشرية والتسويق (بحدود واضحة — انظر §3).

#### هـ) المنصات التقنية

| الطبقة | الحالة |
|--------|--------|
| **Backend** (FastAPI + PostgreSQL) | الوحدات الأساسية (Epics 0–11) مكتملة؛ مزامنة POS دون اتصال قيد الإكمال |
| **واجهة الويب** (React + TypeScript) | الوحدات التشغيلية والمالية والإدارية متاحة |
| **تطبيق الجوال** (Flutter) | حضور وإجازات؛ عميل POS دون اتصال مخطط |

### 2.3 ما يقع ضمن نطاق المشروع الأكاديمي

- تصميم وتحليل متطلبات ERP للتجزئة.
- نمذجة بيانات متكاملة تربط POS والمخزون بـ GL.
- تنفيذ واجهات ويب عربية مع صلاحيات دقيقة (RBAC).
- إثبات جدوى الترحيل المحاسبي الآلي من أحداث حقيقية (بيع، استلام، راتب…).

## 3. الحدود التصميمية والتقنية (Boundaries & Constraints)

### 3.1 حدود المجال (Domain Boundaries)

| داخل النطاق | خارج النطاق (مقصود) |
|-------------|----------------------|
| تجزئة متعددة الفروع | تصنيع ثقيل، تخطيط إنتاج معقد على نطاق MES/ERP صناعي |
| مخزون تجاري + تحويلات | إدارة سلسلة توريد B2B على مستوى مؤسسي شامل |
| محاسبة قيد مزدوج مع ترحيل آلي | بديل كامل لأنظمة ERP مؤسسية ضخمة (مثل SAP) بكل وحداتها |
| تسجيل دفع داخلي للمتجر | بوابات دفع إلكترونية حقيقية (Stripe، مدى، إلخ) |
| OCR أساسي لفواتير الموردين | فوترة إلكترونية حكومية متكاملة كحل مستقل |

### 3.2 حدود تقنية مقصودة

- **الدفع:** يعتمد النظام على مزوّد دفع داخلي (`InStoreLedgerProvider`) لتسجيل النقد والبطاقة والتحويل — **بدون** اتصال ببوابة دفع خارجية؛ وهذا قرار تصميمي يحدّ النطاق ويُبسّط الامتثال في مرحلة التخرج.
- **العمل دون اتصال:** عقود المزامنة مُعرَّفة؛ التنفيذ الكامل لـ PWA وعميل Flutter POS لا يزال قيد الإكمال.
- **التقارير المتقدمة:** قائمة التدفقات النقدية والقوائم المالية المترجمة متعددة العملات — مخططة ولم تُستكمل بالكامل.
- **الأمان:** بعض عناصر التصلية (مثل تخزين refresh token في httpOnly cookie) موثّقة كفجوات قيد المعالجة.

### 3.3 حدود معمارية البرمجيات

```
api/v1/     →  تحقق الطلبات وتوصيل الاستجابات فقط
services/   →  قواعد الأعمال والمعاملات والترحيل إلى GL
models/     →  نماذج قاعدة البيانات (SQLAlchemy)
schemas/    →  عقود الإدخال والإخراج (Pydantic)
```

هذا الفصل يضمن أن مناقشة لجنة التحكيم يمكن أن تركز على **قرارات التحليل والتصميم** دون الخلط بين طبقة العرض وطبقة المنطق.

## 4. الأمن وإدارة المستخدمين (RBAC & Identity)

### 4.1 مبدأ الإدارة المركزية — لا تسجيل مفتوح (No Public Sign-Up)

- **لا يوجد** مسار عام للتسجيل الذاتي (Sign Up) للموظفين أو المستخدمين الإداريين.
- إنشاء حسابات الموظفين يتم **حصرياً** عبر مسؤول مخوّل (صلاحية `users:create`) — عملياً دور **مسؤول IT** (`IT_ADMIN`) أو من يملك صلاحيات إدارة المستخدمين ضمن النظام.
- عند الإنشاء:
  - يُنشأ المستخدم بحالة **معلّقة** (`suspended`) دون كلمة مرور فعّالة.
  - يُطلَق تلقائياً **مهمة انضمام (Onboarding)** لإكمال البيانات من قِبل الموارد البشرية.
  - يُفعَّل الحساب لاحقاً بعد إتمام مسار HR المعتمد.

> **نقطة للجنة التحكيم:** هذا التصميم يعكس نموذج مؤسسي حقيقي — الوصول إلى ERP ليس «خدمة عامة» بل **مورد محمي** تُدار هويته مركزياً.

### 4.2 التحكم في الوصول القائم على الأدوار (RBAC)

- كل مسار API محمي بفحص صلاحية (`require_permission`) على مستوى `(resource, action)`.
- أدوار النظام مُعرَّفة مسبقاً (مثل: `OWNER`, `ACCOUNTANT`, `IT_ADMIN`, `HR_MANAGER`, `WAREHOUSE_MANAGER`, `CASHIER`, `FLOOR_STAFF`, `MARKETING_MANAGER`, `ADMIN`).
- الواجهة الأمامية تُقصّ القائمة الجانبية والمسارات وفق الصلاحيات الفعّالة — لا يُعرض للمستخدم ما لا يملك حق الوصول إليه.

### 4.3 الأدوار المتعددة للمستخدم الواحد (Multiple Roles)

- العلاقة بين المستخدم والدور **many-to-many** عبر جدول `user_roles`.
- يمكن تعيين **أكثر من دور** لنفس المستخدم، مع إمكانية ربط الدور بفرع محدد أو بجميع الفروع (`branch_id = null`).
- **الصلاحيات الفعّالة** = **اتحاد (Union)** صلاحيات جميع الأدوار المعيّنة — ما يتيح سيناريوهات واقعية (مثال: موظف يجمع صلاحيات تشغيلية ومحدودة إدارية).
- لوحة التحكم الرئيسية (`/dashboard`) تُحلّ حسب **أعلى أولوية دور** بين الأدوار المتعددة.

### 4.4 استثناءات موثّقة

- **عملاء POS المؤقتون:** مسار onboarding منفصل للعملاء (ليس للموظفين) عبر رمز مؤقت — خارج نطاق تسجيل المستخدمين الإداريين.
- **حساب Kiosk الحضور:** يُنشأ آلياً لأجهزة الحضور فقط — ليس تسجيلاً مفتوحاً للعامة.
- **حساب المدير الأول (Bootstrap Admin):** محمي بقواعد إضافية لمنع تعطيله أو توسيع أدواره بشكل يهدد استقرار النظام.

### 4.5 مسار الهوية الكامل (للتوضيح أمام اللجنة)

```
IT/Admin ينشئ المستخدم
        ↓
مهمة Onboarding → HR يكمل البيانات والوثائق
        ↓
تفعيل الحساب + تعيين/تعديل الأدوار
        ↓
المستخدم يسجّل الدخول (JWT) بصلاحيات مُجمَّعة من أدواره
        ↓
كل عملية تُسجَّل في Audit Log مع request_id
```

## 5. خلاصة للإلقاء الشفهي (30–60 ثانية)

> **Mezan** نظام ERP للتجزئة متعددة الفروع يجمع نقاط البيع والمخزون والمشتريات والموارد البشرية في منصة واحدة، ويربطها بنواة محاسبية بالقيد المزدوج تُرحَّل إليها القيود آلياً من كل حدث تشغيلي. نفّذنا وحدات Ops وFinance وPeople على Backend FastAPI وواجهة ويب عربية، مع RBAC مركزي لا يسمح بتسجيل مفتوح، ويدعم تعدد الأدوار لكل مستخدم. حددنا بوضوح أن النظام موجّه للتجزئة وليس للتصنيع، وأنه يسجّل المدفوعات داخلياً دون بوابات دفع خارجية — قرار تصميمي يحافظ على تركيز المشروع وقابليته للإثبات أمام لجنة التحكيم.
